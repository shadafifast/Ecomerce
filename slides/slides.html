Latar Belakang & Rumusan Masalah

Latar Belakang

Aplikasi e-commerce seperti Shopee dan Tokopedia melayani jutaan transaksi per hari menggunakan arsitektur Client-Server terdistribusi yang canggih.

Untuk memahaminya secara mendalam, perlu dibangun simulasi nyata yang mencakup: manajemen katalog, keranjang belanja, payment gateway dengan validasi server-side, wishlist, order tracking, serta notifikasi real-time menggunakan Server-Sent Events (SSE).

Setiap fitur dirancang khusus untuk memperlihatkan prinsip Client-Server yang berbeda.
Rumusan, Tujuan & Batasan
  • Rumusan Masalah: Bagaimana arsitektur REST API & SSE berinteraksi dalam mengelola katalog, keranjang, pembayaran, wishlist, dan tracking pesanan secara aman, real-time, dan terkonsistensi?
  • Tujuan:
    • Mengimplementasikan 4 fitur utama e-commerce sebagai demonstrasi prinsip Client-Server.
    • Menerapkan validasi Server-Side untuk mencegah manipulasi data dari Client.
    • Mengimplementasikan SSE sebagai contoh Server Push yang berbeda dari polling REST.
  • Batasan: Implementasi menggunakan Node.js/Express.js dengan database in-memory di lingkungan localhost.

Tinjauan Pustaka Landasan Teori

Kajian Teori Model Client-Server

REST API

REST adalah arsitektur stateless berbasis HTTP. Prinsip: Uniform Interface, pemisahan Client-Server, statelessness, cacheability (Fielding, 2000).

Node.js Event Loop

Node.js menggunakan model asinkron non-blocking I/O dengan event loop tunggal untuk mengelola ribuan koneksi tanpa overhead thread (Tilkov & Vinoski, 2010).

Server-Sent Events

SSE adalah mekanisme Server Push satu arah via HTTP — server mengirimkan event ke client secara real-time tanpa client polling (W3C, 2021).

Keamanan OWASP

Mitigasi: validasi input server-side, pencegahan price-tampering, CORS, dan proteksi XSS pada rendering data dinamis (OWASP, 2023).

Domain E-Commerce

Sistem e-commerce bergantung pada manajemen state keranjang belanja, alur transaksi online, dan sinkronisasi inventori secara real-time (Laudon & Traver, 2021).

REST vs SSE: REST = Client Pull (client meminta data). SSE = Server Push (server aktif kirim event). Keduanya digunakan bersamaan di aplikasi ini.
Payment Gateway: Validasi nominal pembayaran di server-side adalah prinsip terpenting e-commerce — mencegah manipulasi harga oleh client.
Referensi: Laudon & Traver (2021) dikutip dalam Pembahasan (Slide 13) sebagai landasan domain e-commerce: state keranjang, inventori, dan transaksi online.

Payment Gateway — Validasi Nominal Server-Side

Screenshot 2 — Modal Pembayaran: Invoice PENDING + Input Nominal + Preview Kembalian
Alur Payment (2 Fase)
FASE 1 — Checkout (POST /api/checkout)
Server membuat transaksi status PENDING dan mengosongkan keranjang. Stok belum dipotong.
FASE 2 — Payment (POST /api/payment)
Server menerima transactionId + amount. Server memvalidasi: apakah amount ≥ total tagihan. Jika lolos → stok dipotong, status jadi PAID. Jika kurang → Error 400.
Prinsip Keamanan Kritis
  • Harga produk tidak pernah dikirim dari Client — Server menghitung total dari database server.
  • Client tidak dapat memanipulasi total tagihan karena Server memeriksa nilai dari in-memory store-nya sendiri.
  • Validasi nominal pembayaran dilakukan 100% di Server — Client hanya mengirimkan angka.

Wishlist — State di Server

  • Server memvalidasi duplikasi — jika produk sudah ada, Server mengembalikan 409 Conflict.

SSE Live Feed — Server Push
  • Client membuka satu koneksi persisten ke GET /api/events menggunakan EventSource API.
  • Server menyimpan referensi koneksi dan broadcast event ke semua client yang terhubung setiap ada aksi (keranjang, wishlist, pembayaran, tracking).
  • Berbeda dari REST: Tidak ada polling — Server yang aktif mengirimkan data saat event terjadi.
Perbedaan paradigma: REST = Client Pull | SSE = Server Push

Mekanisme API
  • Client mengirim PUT /api/transactions/:id/status untuk memajukan status.
  • Server memvalidasi: pesanan harus berstatus PAID dan belum DELIVERED.
  • Server memperbarui trackingStatus ke tahap berikutnya dalam urutan yang ditentukan server (bukan client).
  • Setelah update, Server broadcast SSE event order_status_updated ke semua client yang terhubung.
Client tidak dapat melompat status (misal dari PENDING ke DELIVERED). Urutan sepenuhnya dikontrol oleh logika Server.

Diagram Alur: Kenapa Payment yang Dibedah?

Sequence Diagram Pembayaran
Fase Checkout
  1. User klik "Checkout Sekarang".
  2. Client kirim POST /api/checkout.
  3. Server validasi stok semua item.
  4. Server buat transaksi PENDING, kosongkan cart.
  5. Server kirim response: ID transaksi + total tagihan.
  6. Client tampilkan Payment Modal.
Fase Payment (Inti Validasi)
  1. User input nominal uang, klik "Bayar".
  2. Client kirim POST /api/payment dengan amount.
  3. Server validasi: amount ≥ total. Jika gagal → Error 400.
  4. Server potong stok, ubah status → PAID.
  5. Server broadcast SSE event ke semua client.
  6. Client terima response kembalian, tampilkan modal sukses.

Aspek Keamanan Server-Side Validation & Proteksi

Aspek Keamanan Client-Server

Price Tampering

Ancaman: Client memodifikasi harga barang di payload HTTP Request sebelum checkout.

✓ Dimitigasi: Server tidak menerima harga dari client sama sekali. Request hanya membawa productId & quantity. Harga dihitung murni dari database server.
Payment Manipulation

Ancaman: Client mengirim nominal pembayaran yang lebih kecil dari total tagihan untuk "membeli gratis".

✓ Dimitigasi: Server memvalidasi amount >= total sebelum memproses. Jika kurang, server mengembalikan HTTP 400 dengan detail kekurangan.
Wishlist Duplikasi

Ancaman: Client mengirim multiple request untuk menambah produk yang sama ke wishlist.

✓ Dimitigasi: Server memeriksa array wishlist dan mengembalikan HTTP 409 Conflict jika produk sudah ada. Client-side cache hanya sebagai UI hint.
⚠️ Status Jumping Attack: Client tidak dapat melompat status tracking pesanan. Server hanya memperbolehkan transisi berurutan: PROCESSING → SHIPPED → DELIVERED. Setiap request divalidasi posisi status saat ini.
⚠️ Keterbatasan: Aplikasi tidak menggunakan HTTPS, autentikasi JWT, atau sanitasi XSS aktif. Data bersifat volatil (in-memory). Ini disebabkan batasan implementasi lokal — pada sistem produksi, ketiganya wajib ada.

Pembahasan Analisis Fitur vs Prinsip Client-Server

Pembahasan: Pemetaan Fitur & Prinsip

Fitur Prinsip Client-Server yang Didemonstrasikan Endpoint Kunci Validasi Server-Side
Katalog & Keranjang Stateless REST — setiap request independen, server menyimpan state GET /api/products, POST /api/cart Cek stok saat tambah keranjang
Wishlist Server sebagai single source of truth untuk state data POST /api/wishlist Validasi duplikasi — HTTP 409
Payment Gateway Server-Side Validation — client tidak dipercaya untuk validasi bisnis POST /api/checkout, POST /api/payment amount ≥ total; harga dari DB server
Order Tracking State Machine di Server — urutan transisi dikontrol server PUT /api/transactions/:id/status Status harus urut, tidak bisa lompat
Live Feed (SSE) Server Push — server aktif mengirim event tanpa request client GET /api/events Koneksi persisten; broadcast ke semua
Stateless REST terbukti efisien untuk operasi CRUD. Selaras dengan Fielding (2000) bahwa pemisahan client-server meningkatkan skalabilitas komponen secara independen.
Single-threaded event loop Node.js menangani SSE persistent connections dengan baik, membuktikan teori Tilkov & Vinoski (2010) tentang efisiensi non-blocking I/O.
Keterbatasan in-memory: Data volatil & tidak ada isolasi antar user. Sistem produksi membutuhkan database persisten dan autentikasi JWT (Laudon & Traver, 2021).

Kesimpulan & Saran Rangkuman Penelitian

Kesimpulan & Saran Pengembangan

Kesimpulan Utama
  1. Arsitektur Client-Server berbasis REST API dengan format pertukaran data JSON sangat andal untuk memisahkan logika presentasi dari logika bisnis server.
  2. Fitur Payment Gateway 2-fase (Checkout PENDING → Payment PAID) berhasil mendemonstrasikan pentingnya validasi bisnis di sisi server, termasuk pencegahan price-tampering dan payment manipulation.
  3. Fitur Wishlist memperlihatkan server sebagai single source of truth dengan validasi duplikasi berbasis HTTP status code semantik (409 Conflict).
  4. Fitur Order Tracking mengilustrasikan konsep state machine yang dikontrol penuh oleh server — client tidak dapat memaksa transisi status yang tidak valid.
  5. Server-Sent Events (SSE) berhasil mengimplementasikan paradigma Server Push, berbeda secara fundamental dari pola Client Pull pada REST biasa.
Saran Pengembangan Lanjut
  • Database Persisten: Integrasikan MySQL/PostgreSQL atau MongoDB agar data tidak hilang saat server restart.
  • Autentikasi JWT: Tambahkan sistem login dengan JSON Web Token agar setiap user memiliki keranjang, wishlist, dan riwayat transaksi yang privat.
  • WebSocket: Tingkatkan SSE ke WebSocket untuk komunikasi dua arah real-time (misal: chat dukungan, notifikasi harga berubah).
  • HTTPS & Rate Limiting: Aktifkan TLS/SSL dan batasi jumlah request per IP untuk keamanan production.
  • Microservices: Pisahkan Payment Service, Catalog Service, dan Notification Service menjadi layanan independen seperti arsitektur Shopee/Tokopedia sesungguhnya.
