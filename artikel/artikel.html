BAB I
PENDAHULUAN

1.1 Latar Belakang

Perkembangan teknologi informasi dan komunikasi dalam satu dekade terakhir telah mengubah lanskap perdagangan global secara fundamental. Pergeseran paradigma dari perdagangan konvensional (offline) menuju sistem perdagangan elektronik (e-commerce) menuntut infrastruktur perangkat lunak yang tangguh, terukur, dan aman. Di balik kemudahan pelanggan menelusuri katalog barang, memasukkan item ke dalam keranjang, dan melakukan pembayaran secara nirkabel, terdapat serangkaian kompleksitas komputasi jaringan yang terjadi di belakang layar (backend).

Aplikasi web modern secara masif mengadopsi arsitektur client-server yang terdistribusi. Antarmuka pengguna (frontend) berjalan di sisi peramban web pelanggan, sementara logika bisnis dan pemrosesan data dieksekusi di sisi server. Komunikasi di antara kedua entitas ini sangat bergantung pada Application Programming Interface (API), khususnya Representational State Transfer (REST). Protokol HTTP dimanfaatkan secara maksimal untuk melakukan transaksi pertukaran data dalam format JSON. Namun, perpisahan komponen fisik ini membawa sejumlah tantangan teknis yang kritis.

Tantangan terbesar bagi para insinyur perangkat lunak dalam sistem e-commerce adalah menjaga integritas dan konsistensi data secara real-time. Kasus umum yang sering memicu kegagalan sistem adalah masalah konkurensi (concurrency) dan race condition. Ketika sebuah toko online mengadakan kampanye penjualan kilat (flash sale), ribuan klien akan mengirimkan permintaan HTTP ke server secara bersamaan untuk membeli produk dengan persediaan terbatas. Apabila logika pengurangan stok barang tidak diimplementasikan dengan cermat, server dapat melayani permintaan yang masuk bersamaan secara redundan, mengakibatkan stok barang di basis data menjadi bernilai negatif. Hal ini disebut fenomena overselling yang berujung pada kerugian finansial maupun reputasi perusahaan.

Di samping masalah skalabilitas beban, keamanan API juga menjadi titik lemah yang krusial. Merujuk pada dokumentasi Open Worldwide Application Security Project (OWASP) Top 10 API Security, celah seperti BOLA (Broken Object Level Authorization) dan Mass Assignment menempati urutan teratas. Seringkali pengembang pemula menaruh kepercayaan penuh pada data yang dikirimkan oleh klien. Sebagai contoh, jika payload JSON dari proses checkout berisi parameter ID Produk, Jumlah, dan Harga, seorang peretas dengan mudah dapat mencegat request tersebut dan memodifikasi harganya menjadi Rp. 1 sebelum masuk ke server. Apabila server hanya menerima dan memproses data tersebut tanpa melakukan validasi otentik ke basis data internalnya (server-side validation), maka eksploitasi transaksi akan terjadi.



Menyadari kedua masalah fundamental di atas, pengembangan arsitektur yang aman memerlukan rancangan khusus. Pemilihan platform Node.js dengan kerangka kerja Express.js menjadi salah satu solusi populer di industri. Node.js memiliki arsitektur single-threaded event loop yang memproses eksekusi sinkron secara bergiliran tanpa interupsi thread. Berdasarkan latar belakang tersebut, laporan ini akan menyajikan perancangan komprehensif mengenai implementasi client-server programming yang utuh.

BAB II
TINJAUAN PUSTAKA

2.1 Konsep Sistem E-Commerce

Electronic Commerce (E-commerce) mengacu pada segala bentuk proses transaksi jual beli produk yang difasilitasi oleh medium internet (Laudon & Traver, 2021). E-commerce mengubah operasional ritel menjadi platform swalayan digital yang dapat diakses 24 jam penuh.

Secara arsitektural, platform belanja modern terdiri dari komponen front-office yang menangani interaksi pengguna dan back-office yang bertugas melakukan rekonsiliasi stok persediaan inventori (inventory sync), memastikan ketersediaan barang fisik di gudang sesuai dengan angka yang tertulis di basis data saat transaksi disahkan pengguna.

2.2 Arsitektur REST API

Representational State Transfer (REST) adalah rancangan gaya arsitektur jaringan yang diciptakan oleh Roy Fielding (Fielding, 2000). REST API menjadi standar untuk komunikasi antar mesin (machine-to-machine).

Kriteria utama REST API meliputi Client-Server Separation (pemisahan klien dan penyimpan data), Statelessness (setiap request harus mandiri tanpa mengingat sesi server), dan Uniform Interface (menggunakan protokol kata kerja HTTP seperti GET, POST, PUT, DELETE secara seragam).

2.3 Node.js dan Single-Threaded Event Loop

Node.js hadir memadukan V8 JavaScript Engine dengan pustaka libuv. Node.js mengadopsi pendekatan Asynchronous Non-Blocking I/O dalam arsitektur Single-Threaded Event Loop. Node.js hanya menggunakan satu thread pemrosesan utama untuk merespons ribuan koneksi (OpenJS Foundation, 2024).

Konsekuensi desain single-thread ini adalah "Atomicity" untuk kode operasi sinkron (synchronous blocking code). Komputasi pengubahan array stok barang tidak akan diinterupsi oleh request pembeli lain sebelum blok fungsinya tuntas.






2.4 Konkurensi dan Race Condition

Konkurensi dalam ilmu komputer adalah kemampuan sistem menangani instruksi bersamaan (Kurose & Ross, 2021). Masalah yang sering terjadi dalam e-commerce adalah "Race Condition".

Race condition terjadi apabila dua proses mencoba membaca nilai basis data lalu memperbaruinya pada waktu berdekatan. Misalnya sisa stok baju adalah 1 buah. Pembeli A dan B menekan beli bersamaan. Jika sistem memvalidasi stok (1 cukup untuk dibeli 1) secara bersamaan, maka hasil akhir stok menjadi negatif. Hal ini memerlukan penanganan khusus di tingkat kode atau basis data.

2.5 Keamanan API (OWASP Top 10)

Laporan "OWASP API Security Top 10" sering dijadikan kiblat standar industri (OWASP Foundation, 2023).

Salah satu parameter ancaman utama adalah "Mass Assignment". Kerentanan ini terjadi saat API memetakan semua parameter payload JSON langsung ke objek internal. Peretas menyusupkan harga fiktif ke dalam request. Pencegahannya menuntut implementasi Server-Side Validation secara mutlak.

BAB V
IMPLEMENTASI SISTEM

5.1 Implementasi Antarmuka Klien (Frontend)

Antarmuka pengguna didesain semenarik mungkin menggunakan CSS khusus yang merender tata letak kotak fleksibel. Layar awal memuat produk secara dinamis melalui panggilan Fetch API.

Gambar 5.1 Implementasi Antarmuka Katalog Utama (Hasil GET /api/products)
// Cuplikan app.js (Sisi Client) - Pengunduhan Data Katalog
async function fetchProducts() {
  try {
    const res = await fetch(`${API_URL}/products`);
    const json = await res.json();
    renderProducts(json.data);
  } catch (error) {
    showToast('Koneksi server gagal!', 'error');
  }
}

Penulisan fetch() mendemonstrasikan bahwa klien bertindak pasif terhadap respons API.


5.2 Implementasi Backend dan Manajemen Keranjang

Manajemen keranjang di-handle murni oleh server (REST stateless pada konteks ini). Ketika pelanggan menambah barang, ID dikirimkan melalui POST request.

Gambar 5.2 Antarmuka Modal Keranjang (Sinkronisasi Endpoint POST /api/cart)
// Mutasi Keranjang - Mengirim Data ke Server (Payload)
async function addToCart(productId, quantity = 1) {
  try {
    const res = await fetch(`${API_URL}/cart`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ productId, quantity })
    });
    const result = await res.json();
    if (res.ok) fetchCart(); // Sinkronisasi ulang DOM dari server
  } catch (e) {
    console.error(e);
  }
}

Middleware express.json() dan ekstensi pertukaran lintas zona (CORS) diinjeksikan agar peramban tidak memblokir akses ke API.


5.3 Mekanisme Atomik dan Mitigasi Keamanan pada Checkout

Bagian krusial yang mendemonstrasikan penanganan eksploitasi parameter harga (OWASP) terdapat pada proses akhir transaksi atau checkout. Gambar berikut menunjukkan keberhasilan operasi checkout.

Gambar 5.3 Implementasi Pencatatan Riwayat Checkout Valid di Server
// Implementasi server.js (Blok Checkout)
app.post('/api/checkout', (req, res) => {
  let itemsToCheckout = [];
  let totalBill = 0;

  // BLOK 1: VALIDASI (Server-Side Pricing & Stock Integrity)
  for (let item of cart) {
    const product = products.find(p => p.id === item.productId);
    if (!product || product.stock < item.quantity) return res.status(400); // Cegah overselling
    
    // Harga (price) ditarik MURNI dari database
    totalBill += product.price * item.quantity;
  }

  // BLOK 2: MUTASI (Pengurangan stok atomik Single-Thread)
  for (let item of cart) {
    const product = products.find(p => p.id === item.productId);
    product.stock -= item.quantity;
  }
  cart = []; res.json({ status: 'success' });
});

Operasi perulangan tanpa I/O asinkron di atas bersifat atomik. Ini mengunci proses hingga stok selesai direduksi.

BAB VII
KESIMPULAN DAN SARAN

7.1 Kesimpulan

Berdasarkan serangkaian studi literatur, perancangan arsitektur logis, penulisan baris kode, hingga hasil simulasi eksekusi terminal server yang dipaparkan, maka dapat ditarik kesimpulan bahwa:

  1. Pendekatan arsitektur perangkat lunak REST API yang diimplementasikan pada backend Node.js (Express) terbukti sangat reliabel untuk menangani komunikasi asinkron stateless antara web klien dan server, terbukti melalui respons endpoint JSON yang cepat tanpa membutuhkan perenderan HTML dinamis dari sisi server.
  2. Perancangan keamanan bisnis yang difokuskan secara otoritatif di sisi backend sukses menutup rapat celah kerentanan pengubahan parameter keamanan (payload tampering). Mekanisme ini memastikan bahwa penetapan harga dan validitas inventori ditarik murni dari database memori lokal aplikasi, terlepas dari apa pun manipulasi permintaan JSON yang ditembakkan klien.
  3. Karakteristik event-loop tunggal (single-threaded) pada V8 Engine Node.js menawarkan pendekatan alamiah untuk mencegah tumpang-tindih perhitungan (race condition). Mekanisme pengurangan stok secara sekuensial array di dalam blok fungsi eksekusi (atomic blocking loop) terbukti mutlak dapat mencegah terjadinya duplikasi pemesanan yang menyebabkan stok minus dalam simulasi skalabilitas jaringan instansi tunggal.

7.2 Saran

Perancangan purwarupa dalam tugas akhir proyek ini merupakan model konseptual dari dasar-dasar Client-Server Programming. Mengingat arsitektur riil industri memegang kerumitan masif, penulis merangkum usulan pengembangan selanjutnya:

  1. Penggantian sistem state memori internal sementara ke layanan sistem database persisten berbasis relasional yang menanamkan teknologi transaksional (ACID), semisal PostgreSQL atau sistem Redis In-Memory khusus untuk lock antrean apabila aplikasi diekspansi ke arsitektur multi-node.
  2. Meningkatkan autentikasi dan kerahasiaan payload dengan melampirkan JSON Web Token (JWT) pada header HTTP, untuk melindungi riwayat pesanan agar eksklusif.
